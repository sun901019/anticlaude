# UX Review 複合技能

## 用途

Pixel 在審核 Dashboard 頁面、新功能 UI、前端改動時調用此技能。
目標：確保每個頁面**可用、清晰、符合 AntiClaude 設計語言**。

---

## 一、AntiClaude Dashboard 設計原則

### 核心設計語言
- **主題**：暖白色系（warm white）+ 中文化介面
- **框架**：Next.js 14 App Router + Tailwind CSS + shadcn/ui
- **字型**：系統字型優先，中文可讀性優先
- **資料視覺化**：Recharts

### 色彩使用
| 用途 | 色彩建議 |
|------|---------|
| 主要行動 | 紫色系（`purple-600`）|
| 成功/已發布 | 綠色系（`green-500`）|
| 警告/待處理 | 琥珀色（`amber-500`）|
| 危險/錯誤 | 紅色系（`red-500`）|
| 中性/空狀態 | 灰色系（`gray-400`）|

## 二、頁面設計審核 Checklist

### 可用性（Usability）
- [ ] 主要功能在 3 秒內找得到
- [ ] 按鈕文字是動詞（「執行分析」不是「分析」）
- [ ] 空狀態（empty state）有說明 + 行動建議
- [ ] Loading 狀態有視覺反饋（spinner 或 skeleton）
- [ ] 錯誤狀態清楚說明原因 + 如何解決

### 資訊層次（Information Hierarchy）
- [ ] 最重要的資訊在頁面上方
- [ ] 數字/指標突顯（大字）
- [ ] 次要資訊視覺上弱化
- [ ] 沒有把所有東西都做得一樣重要

### 一致性（Consistency）
- [ ] 使用現有 shadcn/ui 元件（不重造輪子）
- [ ] 標題層次一致（h1/h2/h3 不跳）
- [ ] 間距遵循 Tailwind 4/8/16/24 的倍數
- [ ] 中文文字不超過每行 40-50 字

### 回應性（Responsiveness）
- [ ] 桌面版正常顯示
- [ ] 表格在窄螢幕有水平滾動
- [ ] 不必硬塞所有資訊進一個畫面

## 三、常見 UI 問題 & 解法

### 問題：表格欄位太多，行動不清楚
**解法**：最後一欄固定為「操作」，用 dropdown 收納次要操作

### 問題：圖表沒有說清楚「所以呢」
**解法**：在圖表下方加一行文字說明關鍵結論

### 問題：頁面太多卡片、沒有主次
**解法**：選出 1 個最重要的指標放大，其他縮小

### 問題：按鈕文字模糊（「確認」、「送出」）
**解法**：改為描述結果的動詞（「發布貼文」、「開始分析」）

### 問題：Empty state 只有「暫無資料」
**解法**：說明為什麼是空的 + 提供下一步（「目前沒有草稿，去選題看看」）

## 四、新頁面設計 SOP

```
1. 確認頁面目標：這個頁面要讓用戶做什麼？
2. 列出需要顯示的資訊（全部）
3. 排優先級：什麼最重要？什麼可以折疊？
4. 選用現有元件（shadcn first）
5. 確認空狀態、loading、錯誤三種狀態都有設計
6. 對照 checklist 審核
7. 交給 Lumi 實作
```
