# E-commerce Pricing, Procurement, and Manual Refinement Spec (2026-03-22)

## Goal

Clarify how the current e-commerce product pricing flow works, identify what should be improved, and define a cleaner UX direction for:

- Product-detail pricing decision
- Cost and logistics estimation
- Inventory inbound / restock flow
- Top-level `Pricing Decision` navigation tab
- Manual / help scope

This spec is intended for implementation on top of the current system. It does **not** require a full rewrite.

---

## 1. Current Behavior: Product Detail -> Pricing Decision

### 1.1 What drives the suggested prices

Inside the product detail drawer, the recommended prices are currently derived from the live merged state of:

- Product base facts
  - `procurement_mode`
  - `cost_rmb`
  - `head_freight_rmb`
  - `weight_kg`
  - `packaging_cost`
  - `is_special_goods`
  - `freight_type`
  - `length_cm`, `width_cm`, `height_cm` for QQL mode
- Shopee scenario inputs
  - `ccb_plan`
  - `is_promo_day`
  - `fulfillment_days`
  - `ad_budget`
- Market reference fields
  - `market_price_high`
  - `market_price_low`
- Product target field
  - `target_price`

The 3 suggested prices shown in the drawer are generated by the local frontend pricing helper and correspond to:

- `traffic`
- `core`
- `profit`

### 1.2 Is it real-time?

Yes, **partially real-time**.

The price cards in the product detail drawer update when the drawer fields change. In practice:

- changing cost / freight / weight / dimensions changes the cards
- changing scenario controls such as CCB / promo / fulfillment also changes the cards
- changing `ad_budget` changes the recommendation

However, the current implementation still mixes:

- product truth
- scenario simulation
- market reference inputs

inside the same decision area, which makes the experience feel unstable and noisy.

### 1.3 Current UX problem

The operator cannot easily tell which of these are:

- immutable product facts
- temporary selling assumptions
- final pricing decisions

This is the main reason the drawer feels messy.

## 2. Current Behavior: Cost / Logistics / System Estimate

### 2.1 Where the estimate comes from

The current estimate comes from two layers:

- Frontend preview logic in `dashboard/src/app/ecommerce/page.tsx`
- Backend canonical pricing logic in `src/ecommerce/router.py` via `calc_full_cost()`

### 2.2 Standard 1688 mode

In standard mode, landed cost is mainly derived from:

- product cost in RMB
- domestic/head freight in RMB
- exchange rate
- selected freight route
  - sea fast
  - sea regular
  - air
- special goods surcharge
- local shipping per kg
- packaging cost

### 2.3 QQL mode

In QQL procurement mode, landed cost is derived from:

- `(cost_rmb + head_freight_rmb) * qql_exchange_rate * (1 + qql_service_fee)`
- billable weight
  - `max(actual_weight, volumetric_weight)`
- `sea_express_rate * billable_weight`
- local shipping and packaging

### 2.4 Current architecture issue

This logic exists in both:

- frontend preview helpers
- backend canonical calculator

That means:

- results can drift
- maintenance cost is higher
- trust decreases when UI and API do not fully match

### 2.5 Required improvement

The backend `calc_full_cost()` must become the only source of truth for:

- landed cost
- fee bundle
- suggested role
- price recommendations
- profitability

The frontend should only:

- collect inputs
- request/re-render estimates
- present results clearly

## 3. Product Detail UX Refinement

### 3.1 Required information hierarchy

The product detail drawer should be split into 3 layers:

#### Layer A: Executive Summary

Always visible at the top:

- product name
- SKU
- current role
- current status
- current stock
- AI summary
- top warning if any

#### Layer B: Decision Cards

Primary operator surface:

- suggested traffic price
- suggested core price
- suggested profit price
- current target price
- estimated gross margin
- current stock risk
- next recommended action

#### Layer C: Expandable Work Areas

Editable sections:

- Basic Product Info
- Cost and Logistics
- Pricing and Market
- Shopee Selling Scenario
- Performance
- Notes / History

### 3.2 What should move out of the default pricing view

Do not keep all of these expanded by default:

- CCB buttons
- promo day toggle
- fulfillment days toggle
- market low/high inputs
- ad budget

These should move into:

- a collapsed `Scenario Simulation` area
- or a sub-section inside `Pricing and Market`

### 3.3 What should stay visible by default

Default visible area should focus on:

- what it costs
- what the system recommends
- whether the product is viable
- what to do next

## 4. Inbound / Restock Flow Review

### 4.1 What works now

The current inbound modal already has:

- SKU selection
- RMB cost input
- quantity input
- landed unit preview
- projected stock
- simple ETA / lead days

This is a good base.

### 4.2 What still feels weak

It still behaves more like a simple stock-entry dialog than a real procurement decision surface.

Missing or weak points:

- no explicit procurement route choice
- no clear route-dependent landed-cost comparison
- no clear inventory coverage impact
- no quick answer to "is this batch worth buying now?"
- no link to the product's current pricing/margin context

### 4.3 Recommended improvements

Add these fields or outputs:

- procurement route
  - standard 1688
  - QQL
- freight route if applicable
- ETA date preview
- projected days of stock after inbound
- landed cost delta vs current product landed cost
- target stock level reference
- one-line recommendation
  - under-buying
  - normal replenishment
  - aggressive restock

### 4.4 UX principle

Inbound is not just stock entry.
It should answer:

- how much will this batch really cost
- what stock level it creates
- whether this is safe and sensible

## 5. Top Navigation: `Pricing Decision`

### 5.1 Current purpose

The top-level `Pricing Decision` tab currently acts as a focused pricing simulator.

It lets the operator:

- choose a SKU
- inspect pricing breakdown
- inspect total cost
- inspect profit / margin
- inspect next action and suggested role

### 5.2 Is it necessary?

It is useful **only if** the team wants a dedicated pricing workspace separate from the product detail drawer.

It becomes redundant if product detail already provides:

- clear price cards
- cost breakdown
- scenario simulation
- role-fit and warnings

### 5.3 Recommended decision

Keep it if the goal is:

- focused operator pricing work
- SKU-by-SKU price simulation
- scenario comparison without entering product edit mode

De-emphasize or merge it if the goal is:

- simpler navigation
- one primary workspace per product

### 5.4 Preferred direction

Recommended direction:

- keep the capability
- reduce its prominence in main nav
- rename or reposition it as:
  - `Pricing Lab`
  - or `Pricing Simulation`

This makes it clear that it is a focused tool, not a duplicate of product detail.

## 6. Manual / Help Scope

### 6.1 Current issue

The current manual/help area contains broader workflow and AI operations content mixed with e-commerce operations.

This is not aligned with the operator's mental model for the e-commerce module.

### 6.2 Required scope

The manual inside the e-commerce area should contain **e-commerce-only** content:

- product intake
- procurement / inbound
- pricing logic
- margin thresholds
- role meanings
- Shopee fee terms
- QQL / standard sourcing explanation
- performance interpretation
- restock recommendation logic

### 6.3 Remove from e-commerce manual

Move out any non-e-commerce content, such as:

- general AI office workflow
- media/content pipeline SOP
- cross-agent discussions not directly tied to e-commerce operation

### 6.4 Recommended manual structure

Use sections:

1. Quick Start
2. Product Intake
3. Pricing Logic
4. Procurement / Inbound
5. Performance and Decisions
6. Shopee Terms Glossary
7. FAQ

## 7. Duplicate / Confusing Areas to Clean Up

### 7.1 Pricing formula duplication

Problem:

- frontend computes recommendation logic
- backend computes recommendation logic

Required fix:

- backend is authoritative
- frontend displays API result only

### 7.2 Mixed configuration ownership

Problem:

- product fact
- selling scenario
- system defaults

are still mixed across quick add, drawer, and settings.

Required fix:

- product facts belong to product
- simulation belongs to pricing view
- defaults belong to settings

### 7.3 Manual overload

Problem:

- too much explanation on operational screens

Required fix:

- short contextual hints on work screens
- full help in manual only

---

## 8. Recommended Implementation Order

1. Product detail pricing section
   - separate summary / decision / advanced simulation
2. Move pricing source of truth fully to backend
3. Improve inbound modal with procurement decision context
4. Narrow manual to e-commerce content only
5. Decide whether top-level `Pricing Decision` remains primary nav or becomes a secondary tool

---

## 9. Final Product Direction

The operator should experience the e-commerce system like this:

- Quick Add creates products quickly
- Product Detail is the main product workspace
- Pricing Decision is either a focused simulation lab or merged into product detail
- Inbound is a replenishment decision tool, not just a stock entry form
- Manual is clean and e-commerce-only

The system should feel:

- professional
- calm
- easy to trust
- operationally efficient

Not:

- duplicated
- noisy
- over-explained
- full of mixed concerns
